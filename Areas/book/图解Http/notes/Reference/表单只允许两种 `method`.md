传统的 HTML 表单（`<form>` 标签）在浏览器中的原生支持只允许两种 `method` 属性值：
- `GET`
- `POST`
浏览器**无法直接**通过标准 HTML 表单发出 `PUT` 或 `DELETE` 请求。因此，如果开发者需要实现更新或删除功能，就必须通过 `POST` 来“伪造”这些请求。
#### 为什么有这种限制
**向后兼容性（Backward Compatibility）** 和 **规范的初衷**：
##### 1. 向后兼容性 (The Biggest Reason)

Web 是一个巨大的、全球性的系统，**兼容性是至高无上的**。

- HTML 规范（包括 `<form>` 标签）是在 90 年代初设计的。当时，`GET` 用于获取文档，`POST` 用于提交数据。
- 如果 HTML 规范突然修改，允许 `<form method="PUT">`，那么全球数以百万计的旧式服务器、代理服务器、防火墙和缓存系统都需要更新才能正确理解和处理这个新的表单方法。这在实践中几乎是不可能的。
- 为了避免打破旧网站，W3C（万维网联盟）选择**不改变**基础的表单规范。
##### 2. 语义和浏览器默认行为的简化
- **表单的本质是导航或数据提交：** 表单的设计目标是**简单的用户交互：要么是从服务器获取一个新页面（`GET`），要么是向服务器发送数据并期待一个状态改变（`POST`）**。
- `PUT` 和 `DELETE` 是幂等（可重复执行且结果一致）的操作，它们的语义更精确，但对于用户界面的简单操作（比如用户点击“提交”）来说，`POST` 这种**不幂等但通用的提交**已经足够，**后续的安全和路由工作都交给服务器端的应用代码去完成**。
