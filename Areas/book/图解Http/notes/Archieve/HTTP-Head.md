HEAD方法和GET方法一样，只是不返回报文主体部分。用**于确认URI的有效性及资源更新的日期时间等**。

图：和GET一样，但不返回报文主体

### 1. 核心特性：只返回头部，不返回主体
- **`HEAD` 请求**与对应的 `GET` 请求所获得的**响应头部**（Response Headers）是完全相同的。
- 唯一的区别是，`HEAD` 响应**永远不会包含响应主体（Body）**。
### 2. 主要用途（你提到的）：

- **确认 URI 的有效性（Resource Availability）：** 通过检查响应的状态码（例如 `200 OK` 或 `404 Not Found`），客户端可以确定请求的资源是否存在且可访问，而无需下载整个资源。
- **确认资源更新的日期时间（Caching）：** 通过检查响应头部中的缓存相关字段，例如：
    - **`Last-Modified`：** 获取资源的最后修改时间。
    - **`Content-Length`：** 获取资源的大小（即使不下载也可以知道）。
    - **`ETag`：** 获取资源的实体标签（用于更精确的缓存校验）。

### 3. 实际益处：

`HEAD` 方法的主要价值在于**高效**。它允许客户端在下载资源之前，先获取元数据（Metadata），从而进行决策，例如：
- **缓存校验：** 客户端可以使用 `Last-Modified` 或 `ETag` 信息，配合 `If-Modified-Since` 或 `If-None-Match` 等条件请求，来判断本地缓存是否仍然有效，从而避免不必要的 `GET` 请求。
- **链接检查：** 网站爬虫或链接检查工具可以使用 `HEAD` 来快速验证大量链接的有效性，而不需要消耗大量带宽去下载实际内容。
