## 3. 展开说明  
### 破坏性变更提交示例（Conventional Commits 风格）

下面展示几种常见的 **破坏性变更（Breaking Change）** 写法：

---

#### 1. 使用 `!` 标记破坏性变更

```text
feat!: 修改用户登录接口返回值结构
```

- **解析**：
    
    - `feat` → 功能类
        
    - `!` → 破坏性变更
        
    - 描述 → “修改用户登录接口返回值结构”
        
- **作用**：自动提示下一个版本需要 **Major 升级**
    

```text
fix!: 修复支付接口返回错误码逻辑
```

- 修复类提交，但破坏了原有接口规范
    
- changelog 会标记为破坏性修复
    

---

#### 2. 使用 `BREAKING CHANGE:` 注释

```text
feat: 改造用户注册流程
BREAKING CHANGE: 注册接口由 POST /user 改为 POST /account
```

- **解析**：
    
    - 前缀 `feat` → 功能
        
    - 换行后的 `BREAKING CHANGE:` → 明确告知破坏性变更
        
    - 描述 → 详细说明改动影响
        

```text
fix: 修复订单创建逻辑
BREAKING CHANGE: 订单金额字段由 number 改为 string
```

- 即使是 bug 修复，如果改变了 API 类型，也属于破坏性变更
    

---

#### 3. 带 scope 的破坏性变更

```text
feat(auth)!: 重构认证模块
BREAKING CHANGE: JWT 生成方式改变，旧 token 无法使用
```

- `auth` → 范围（scope）
    
- `!` → 破坏性变更
    
- BREAKING CHANGE 描述具体影响
    

---

#### 总结规律

1. **两种标记方式**：
    
    - `!` 放在 type 后
        
    - 或者 `BREAKING CHANGE:` 后跟详细说明
        
2. **changelog 工具**：
    
    - 会自动识别并分类到 **破坏性变更** 或 **Major 升级**
        
3. **可选 scope**：
    
    - 更清晰地说明影响模块
