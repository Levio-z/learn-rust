


除非另有说明，否则所有标准 HTTP 行为都应由客户端和服务器承担。这包括（但不一定限于）：
### 没有存储库 404
如果 `$GIT_URL` 上没有存储库，或者与 `$GIT_URL` 匹配的位置指向的资源不存在，则服务器不得响应 `200` `OK` 响应。 服务器应该响应 `404` `Not` `Found`、`410` `Gone`，或任何其他不意味着资源按请求存在的合适 HTTP 状态代码。
### 不允许访问 403
如果 `$GIT_URL` 上有一个存储库，但当前不允许访问，则服务器必须使用 `403` `Forbidden` HTTP 状态代码进行响应。

服务器应同时支持 HTTP 1.0 和 HTTP 1.1。服务器应支持请求和响应正文的分块编码。


服务器可以返回 ETag 和/或 Last-Modified 标头。

客户端可以通过包含 If-Modified-Since 和/或 If-None-Match 请求标头来重新验证缓存的实体。

如果出现相关标头，服务器可以返回 `304` `未``修改` ，并且实体没有更改。 客户必须治疗 `304` `未``修改` 与 `200` 相同 通过重用缓存的实体 OK。

如果 Cache-Control 和/或 Expires 标头允许缓存，则客户端可以重用缓存的实体而无需重新验证。客户端和服务器必须遵循 RFC 2616 的缓存控制。
