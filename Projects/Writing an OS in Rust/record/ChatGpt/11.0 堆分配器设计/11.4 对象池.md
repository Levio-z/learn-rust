对象池设计模式创建**一组可以重用的对象**。当需要新对象时，会从池中请求该对象。如果先前准备好的对象可用，则立即返回该对象，从而**避免实例化成本**。如果池中不存在对象，则会创建并返回新项。当对象已被使用并且不再需要时，它将返回到池中，以便将来再次使用它，而无需重复计算成本高昂的实例化过程。需要注意的是，一旦对象被使用并返回，现有引用将失效。
在某些对象池中，资源是有限的，因此指定了最大对象数。如果达到此数字并请求新项，则可能会引发异常，或者线程将被阻止，直到将对象释放回池中。
对象池设计模式在 .NET Framework 标准类的多个位置使用。一个示例是 .NET Framework Data Provider for SQL Server。由于 SQL Server 数据库连接的创建速度可能很慢，因此会维护一个连接池。关闭连接实际上不会放弃与 SQL Server 的链接。相反，连接保存在池中，请求新连接时可以从池中检索它。这大大提高了建立连接的速度。

### 好处
**在初始化类实例的成本很高且类的实例化和销毁率高的情况下，对象池可以提供显着的性能提升**——在这种情况下，对象可以经常重用，并且每次重用都会节省大量时间。对象池需要资源——内存和可能的其他资源，例如网络套接字，因此最好在任何时候使用的实例数量较少，但这不是必需的。

池化对象是在可预测的时间内获得的，**而创建新对象（尤其是通过网络）可能需要可变的时间。这些好处主要适用于时间成本高昂的对象，例如数据库连接、套接字连接、线程和大型图形对象**，如字体或位图。

在其他情况下，简单的对象池（不持有外部资源，而仅占用内存）可能效率不高，并且可能会降低性能。[[1]](https://en.wikipedia.org/wiki/Object_pool_pattern#cite_note-urban-1) 在简单的内存池的情况下， [板分配](https://en.wikipedia.org/wiki/Slab_allocation "Slab allocation")内存管理技术更合适，因为唯一的目标是通过减少碎片来最大限度地降低内存分配和释放的成本。
