## 1. 核心观点  

**核心结论**：上下文切换是一项昂贵操作，不仅消耗 CPU 时间，还可能降低缓存效率和系统吞吐量，尤其在锁争用或多线程密集切换场景中成本更高。

## 3. 展开说明  
**保存和恢复的信息**
- CPU 寄存器
	- 保存当前线程的执行状态，保证恢复后能继续运行
- 操作系统记录
	- 线程/进程控制块（TCB / PCB）
		- 优先级、状态、调度信息等
- 缓存与高速缓冲
	- CPU 缓存可能**含有被暂停线程使用的数据**
	- 切换到另一个线程后，缓存命中率下降，需要重新加载（cache miss）


**成本**
- 保存和恢复上下文的CPU 指令开销：主要开销是CPU执行指令的周期的开销：保存/写入寄存器、切换栈指针、更新 PCB。
	- **CPU 寄存器**
	    - 保存通用寄存器、程序计数器（PC）、栈指针（SP）、标志寄存器等
	- **内核栈状态**
	    - 保存当前线程的内核栈，以便返回时恢复现场
	- **线程控制块（TCB / PCB）**
	    - 写入线程状态、优先级、调度信息等
- 缓存失效成本：主要开销是CPU等待数据的周期，跨核通信等待周期。
	- 切换线程可能导致缓存行被替换 → 影响性能
		-  新线程访问的数据可能不在 CPU 缓存 → Cache miss
		- CPU pipeline 可能被刷新 → 指令流水线损失
		- TLB（虚拟内存映射缓存）不同进程：TLB 大量失效 → 页表访问延迟
	- （共享锁可能导致缓存行在多个核间同步 → 延迟增加）

不确定是不是上下文切换的开销
- 操作系统管理开销
	- 调度开销
		- 操作系统运行调度算法，选择下一个可运行线程
	- 处理同步原语（Mutex、Semaphore、Condition Variable）
		-  如果当前线程释放锁或信号量，可能唤醒阻塞线程
	    - 阻塞/唤醒过程本身可能触发更多上下文切换
		- 可能涉及优先级调整、死锁检测、信号处理等
